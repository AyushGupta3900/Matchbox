# Business Idea — What MatchBox Is (in plain language)

> Status: **Reference** · The "explain it to a human" doc. Read this before anything else.
> Everything technical ([01-requirements-and-scope.md](01-requirements-and-scope.md) onward)
> is just this idea, made precise.

## What is a trading exchange?

Imagine a giant marketplace where people buy and sell one thing — say Bitcoin — using
dollars.

- Some people want to **buy** Bitcoin ("I'll pay $60,000 for 1 BTC").
- Some people want to **sell** Bitcoin ("I'll sell 1 BTC for $60,000").

The exchange's only job is to be the **matchmaker** in the middle. When a buyer's price
meets a seller's price, it **matches** them, the trade happens, and money + Bitcoin change
hands.

That's the whole product. Binance, Coinbase, the New York Stock Exchange — at their core,
they all do this one thing. We're building that core. Ours is called **MatchBox**.

## The order book — the heart of it

When you say "I want to buy 1 BTC at $60,000," that's an **order**. If nobody is selling at
that price right now, your order doesn't vanish — it *waits* in a list. That waiting list
of all unmatched orders is the **order book**.

Picture two queues at a counter:

```
        SELLERS (asks)              ← people willing to sell, cheapest first
        $60,100  →  2 BTC
        $60,050  →  1 BTC
        $60,010  →  5 BTC   ← best (lowest) ask
        ─────────────────  ← "the spread" (the gap between best buy and best sell)
        $60,000  →  3 BTC   ← best (highest) bid
        $59,990  →  1 BTC
        BUYERS (bids)              ← people willing to buy, highest first
```

A new buyer willing to pay **$60,010** matches the cheapest seller at $60,010. Trade done.
The matched orders leave the book.

The rule for who matches first is **price-time priority**:
1. **Best price wins** (highest bid / lowest ask).
2. Same price → **whoever was there first** wins (a fair queue).

Building this matching logic correctly *and* fast is the main challenge of the project.

## Why this is hard (and worth building)

Matching sounds easy. Doing it **correctly** and **fast** is not, because:

- **Money must never be wrong.** If two buyers grab the last seller at the same instant,
  you can't sell the same Bitcoin twice. Every balance stays exact, to the penny.
- **Speed matters enormously.** Real exchanges match in *microseconds*. Traders care about
  the slowest 1% of responses (the "tail"), not the average.
- **You must never lose history.** If the server crashes mid-trade, you must rebuild
  *exactly* what happened. No lost trades.

Correctness + speed + never-lose-data are exactly the problems senior backend engineers get
paid to solve. That's why this project was chosen — it forces the hard versions of those
skills.

## What we're going to build (in order)

The project is split into phases. Each phase adds one capability and deliberately *creates a
problem* that the next phase solves.

| Phase | What it adds | The big idea you learn |
|-------|-------------|------------------------|
| **0** | Accounts + wallet (deposit money, track balances) | Double-entry accounting — money is only moved, never created or destroyed |
| **1** | The matching engine (order book + matching) | A data structure that *is* the system; making it deterministic |
| **2** | Event sourcing (store every change as history) | The *list of events* is the truth, so you can replay and recover |
| **3** | Live market data over WebSockets | Streaming to thousands of subscribers; caching |
| **4** | Handle huge concurrent load | Lock-free concurrency — one thread owns the book, no traffic jams |
| **5** | Make it extremely low-latency | Memory/GC tuning — keeping the slowest 0.1% fast |
| **6** | Observability + nightly checks | Measuring p99 latency; proving the books balance nightly |

We start at **Phase 0** and resist jumping ahead.

## The four "always true" rules we live by

1. **Money is whole numbers, never decimals.** Computers fumble decimal math
   (0.1 + 0.2 ≠ 0.3 in code). So we count in the smallest unit — cents and satoshis — using
   integers. No rounding bugs.
2. **One worker owns the matching.** Instead of many threads fighting over the order book
   (and corrupting it), exactly *one* thread does all matching. Simpler and, surprisingly,
   faster.
3. **Check funds before matching, never during.** We "reserve" your money the moment you
   place an order, so the fast matching engine never pauses to ask the database "can they
   afford this?"
4. **The database is a copy, not the original.** The real truth is the stream of events
   ("order placed," "trade happened"). The database is rebuilt from that stream.

## Who uses it

- **Traders** — place and cancel orders, deposit and withdraw money.
- **Market-data consumers** — bots/screens watching live prices and trades.
- **Internal systems** — settlement (moving balances), analytics, surveillance — these read
  the event stream, not the public API.
- **Ops/admin** — run reconciliation, halt a market, inspect health.

## In one sentence

> MatchBox is the matchmaking engine at the core of a crypto exchange: it takes buy and sell
> orders, matches them fairly by price and time, keeps everyone's balance exact, and remembers
> everything that ever happened — correctly, quickly, and without losing data.
